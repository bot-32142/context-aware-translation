3. RESOURCE CLAIM CONFLICT DETECTION
================================================================================

Task 1: Translate docs [1, 2]                Task 2: Translate docs [2, 3]
        ↓                                             ↓
Claims: {                                    Claims: {
  ResourceClaim("doc", "book-x", "1"),         ResourceClaim("doc", "book-x", "2"),
  ResourceClaim("doc", "book-x", "2"),         ResourceClaim("doc", "book-x", "3"),
}                                            }

                    ↓
        ClaimArbiter.conflicts(claims1, claims2)
                    ↓
        Checks: ("doc", "book-x") matches
                key="1" vs "2" → overlap? YES
                key="2" vs "2" → overlap? YES ✗ CONFLICT
                key="2" vs "3" → overlap? YES

Result: Cannot run Task 2 while Task 1 is running
        ✓ Task 1 must complete first
        ✓ TaskEngine prevents simultaneous launch


4. STAGE PIPELINE EXECUTION FLOW
================================================================================

For each stage (translation or polish):

    ┌──────────────────────────────────────────────────────────────┐
    │  1. SUBMIT: Build requests, upload to provider (batches)    │
    └──────────────────┬───────────────────────────────────────────┘
                       ↓
    ┌──────────────────────────────────────────────────────────────┐
    │  2. POLL: Check batch status until terminal (COMPLETED/FAILED)│
    │     - Long poll with exponential backoff                      │
    │     - Check cancel_requested flag periodically                │
    │     - 24-hour timeout per batch                               │
    └──────────────────┬───────────────────────────────────────────┘
                       ↓
    ┌──────────────────────────────────────────────────────────────┐
    │  3. VALIDATE: Deserialize responses, validate JSON          │
    │     - For each item: llm_batch_store.get_completed_response()│
    │     - validated_chat() to ensure response matches schema     │
    │     - Mark item as "succeeded" or "failed"                   │
    └──────────────────┬───────────────────────────────────────────┘
                       ↓
    ┌──────────────────────────────────────────────────────────────┐
    │  4. FALLBACK: Retry failed items via direct LLM call        │
    │     - NOT via batch (faster feedback)                        │
    │     - Same validation logic (validated_chat)                 │
    │     - Only if translation succeeded (polish only)            │
    │     - Mark as "succeeded" or leave "failed"                  │
    └──────────────────┬───────────────────────────────────────────┘
                       ↓
                   [continue to next stage
                    or apply results]

7. HANDLER PROTOCOL (TaskTypeHandler)
================================================================================

Implemented by: BatchTranslationHandler

Methods:

  task_type: str
    ↓
    Identifier used by TaskEngine.register_handler(handler)

  decode_payload(record: TaskRecord) → dict
    ↓
    Deserialize JSON from record.payload_json

  scope(record, payload) → DocumentScope
    ↓
    Returns: AllDocuments | SomeDocuments | NoDocuments

  claims(record, payload) → frozenset[ResourceClaim]
    ↓
    Returns resource claims (used for conflict detection)

  can(action, record, payload, snapshot) → Decision
    ↓
    Policy check: is this action allowed given current state?
    action = RUN | CANCEL | DELETE
    snapshot = ActionSnapshot (running_task_ids, active_claims, now, retry_after)

  can_autorun(record, payload, snapshot) → Decision
    ↓
    Autorun eligibility: should TaskEngine.tick() start this task?

  build_worker(action, record, payload, deps) → Worker
    ↓
    Factory: creates BatchTranslationTaskWorker configured for action


8. ERROR HANDLING CHAIN
================================================================================

During run_task():

    ┌─ Transient Error (network, timeout, rate limit)
    │  ├─ _is_transient_batch_error(exc) → True
    │  ├─ Mark status=PAUSED, last_error=str(exc)
    │  └─ Next tick() can retry
    │
    ├─ Permanent Error (invalid config, schema mismatch)
    │  ├─ _is_transient_batch_error(exc) → False
    │  ├─ Mark status=FAILED, phase=PHASE_DONE
    │  └─ Manual rerun only
    │
    ├─ _PauseRequestedError (local thread interrupted)
    │  ├─ Check if cancel_requested flag set in DB
    │  ├─ If yes: call request_cancel() → CANCELLING/CANCELLED
    │  └─ If no: status=PAUSED (resume on next tick)
    │
    └─ OperationCancelledError (core.cancellation)
       └─ Same as _PauseRequestedError


9. DECISION WORKFLOW (UI → Action)
================================================================================

User clicks button
  ↓
UI calls task_engine.preflight(task_type, book_id, params, action)
  ├─ Builds draft TaskRecord
  ├─ Calls handler.can(action, record, payload, snapshot)
  ├─ Checks claim conflicts via ClaimArbiter
  └─ Returns Decision(allowed: bool, reason: str)
  ↓
If Decision.allowed:
  ├─ Button shows: ENABLED, tooltip=""
  └─ User can click → task_engine.submit/cancel/delete()
Else:
  ├─ Button shows: DISABLED, tooltip=Decision.reason
  └─ User cannot click
